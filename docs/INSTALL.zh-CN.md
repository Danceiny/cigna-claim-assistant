# Cigna Claim Assistant 安装与使用清单

## 发布包里有什么

- `extension/`：Chrome 插件形态，负责复用已登录 Cigna 页面并自动提交。
- `cigna-claim-assistant-extension.zip`：Chrome 扩展压缩包，方便归档。
- `cigna-claim-assistant-utools.upx`：uTools 插件形态，负责桌面入口、本地目录扫描和提交计划导出。
- `START.html` 和 `OPEN_INSTALLER.command`：安装入口和 macOS 辅助打开脚本。
- `scripts/` 和 `src/`：本地 OCR sidecar、扫描、压缩和审计 helper。

总 release 包只是交付容器；实际产品形态是 Chrome 插件和 uTools 插件。真实 Cigna 自动提交依赖 Chrome 登录态，因此由 Chrome 插件完成。

## 安装 Chrome 插件

1. 解压 `cigna-claim-assistant-release.zip`。
2. 打开 Chrome 的 `chrome://extensions`。
3. 打开右上角 `Developer mode`。
4. 点击 `Load unpacked`。
5. 选择解压目录里的 `extension` 文件夹。
6. 在同一个 Chrome 里登录 Cigna Envoy。

不要用临时 Chrome profile 安装；扩展会复用当前 Chrome 的登录态。

## 基础设置

打开扩展后先确认：

- 被保险人姓名：要和 Cigna 页面卡片上的姓名一致。
- 报销理由：例如 `BACK_PAIN`；没有内置默认值，必须明确填写或从备份导入。
- 治疗国家/地区：默认 `阿拉伯联合酋长国`，会用于 Cigna 下拉框选择。
- 理赔类型：默认 `医疗类`。
- 就诊类型：默认 `门诊`。
- 付款账户关键词：例如 `BANK 0001`；付款页必须出现并能匹配该账户才会继续。
- 长期病症最早治疗日期：可选兜底值。扩展会优先使用 PDF/OCR/人工修正里的最早治疗日期点击 Cigna 日期选择器，不手动输入；它不是服务日期，也不会拿服务日期冒充最早治疗日期。
- 最早提交服务日期：可留空。不填则不限制服务日期；填写后，早于该日期的材料会阻塞。
- 是否允许提交 `review` 项：默认允许普通扫描件类 `review`；取消后只提交 `ready`。

无法识别文件类型或低置信度的 `review` 会被要求人工确认，即使勾选允许 review 也不会自动提交。

提交前扩展会在本地检查必填项，并确认每单都有最早治疗日期。缺失时不会发送提交请求，修正后可直接再次点击 `自动处理并提交`。

设置区下方会显示自动提交是否已就绪；缺少被保险人、报销理由、付款账户关键词等必填项时，会直接列出缺项。

设置区下方也会显示防重记录摘要，包括本地已记录的提交日期、理赔键和文件 hash 数。导入记录、导入备份、清空记录或提交成功后，这个摘要会刷新。

## 推荐流程

1. 先在 Chrome 里打开 Cigna 并确认已登录；不必手动停在新建理赔页，扩展会复用已登录标签页并导航到中文新建理赔入口。
2. 点击扩展里的 `检查 Cigna 页面`。如果当前 Cigna 标签页不在新建理赔入口，扩展会自动切到该入口后再检查。
3. 大批文件建议使用 Chrome 侧边栏打开 Cigna Claim Assistant，或点击 `独立页面` 在完整标签页里拖拽、修正和压缩，避免 popup 失焦关闭。
4. 把报销 PDF/图片或整个文件夹拖进扩展，或点击 `选择文件夹，或拖入 PDF/图片/文件夹`；直接拖入文件夹时会递归读取其中的 PDF/图片，直接拖入文件时只加入这些文件。
5. 点击 `识别文件`。
6. 看 summary：
   - `将提交` 是会提交的日期数。
   - `将跳过` 是 blocked 或被设置跳过的日期数。
   - `需压缩` 必须先处理到 0。
7. 如果扫描件识别错，在文件行里修正：
   - `类型`：理赔表/医疗文件、发票、收据等。
   - `服务日期`：用于一天一单分组。
   - `最早治疗日期`：用于 Cigna 日期选择器；通常可留空使用基础设置里的长期病症日期。
   - 缩略图只用于本地核对，不会进入账本或导出的计划文件。
   - 选错文件时点 `移除`，当前计划会立即重新计算。
   - 整批选错时点 `清空批次`，不会清除基础设置和提交记录。
8. 点击 `导出计划`，快速检查日期、缺失文件、压缩计划、重复项和最近一次提交快照。
9. 点击 `提交预检`，下载将提交的日期、字段、附件名、大小和 hash 清单；预检不会启动 Cigna 提交，也不会导出 PDF/图片内容。
10. 可选：点击 `真实页面彩排`，在真实 Cigna 页面走到最终检查页后停止。彩排会真实上传附件并选择付款账户，但不会勾选免责声明、不会点击最终提交，也不会写入防重账本。
11. 需要留存页面验收证据时，点击 `导出 Cigna 诊断` 下载只读诊断 JSON；它只检查当前 Chrome 登录态里的 Cigna 页面，不上传文件、不提交。
12. 点击 `自动处理并提交`。
13. 提交完成后点击 `导出备份` 保存基础设置、本地账本、最近一次提交快照和最近一次 Cigna 诊断；只想保存防重账本时可点 `导出记录`。

`拖入/选择文件后自动处理并提交` 默认开启；选择文件夹、添加文件或拖拽文件都会自动执行识别、压缩、防重检查、Cigna 页面预检和提交。缺少必填设置、仍需压缩、无法确定最早治疗日期、只有发票或已有提交进行中时，扩展只会显示错误或跳过原因，不会进入 Cigna 提交动作。需要先人工看计划时，可临时关闭这个开关，再手动点击 `自动处理并提交`。

`真实页面彩排` 和 `提交预检` 不一样：提交预检只在本地导出 JSON，不触碰 Cigna 表单；真实页面彩排会操作已登录 Cigna 页面并上传附件，只是在最终提交动作前停止。

提交成功后，扩展会立即按最新账本刷新当前批次；已经成功的日期会从可提交列表里移除，避免重复提交。

计划和提交预检都会显示最早治疗日期来源：PDF/OCR 文本、人工修正或全局兜底。提交前重点确认每单日期来源，避免把服务日期或错误兜底日期提交到 Cigna。全局兜底只会在整单没有 PDF/OCR/人工修正日期时使用；如果某个相关 PDF/OCR 文本识别出了最早治疗日期，它会优先于全局兜底。

扫描 PDF 没有文本层时，扩展会继续使用文件名、日期文件夹、`scan_1/scan_2` 规则和人工修正。已有 OCR 文本时，把同名 `scan_1.pdf.txt` 或 `scan_1.txt` 一起拖入扩展；这些文本只用于识别，不会作为报销附件上传。

可以先批量生成扩展可用的 OCR sidecar：

```bash
npm run claims:ocr-sidecars -- --dir /Users/bytedance/Documents/报销 --ocr-command /path/to/ocr-wrapper
```

如果是在解压后的 release 包里运行本地 helper，先进入 release 根目录执行一次 `npm install`。
需要确认本地 helper 可运行时，可继续执行 `npm run claims:test:helper`；它会检查核心同步并用默认输出路径跑一次空目录扫描。

未传 `--ocr-command` 时会尝试 `tesseract <file> stdout -l eng`。生成结果默认是 `原文件名.pdf.txt`，扩展拖入原文件夹时会自动读取。已有 sidecar 默认不会覆盖；需要重跑时加 `--overwrite`。也可以用本地扫描 CLI：`npm run claims:scan -- --dir /Users/bytedance/Documents/报销 --ocr --ocr-command /path/to/ocr-wrapper` 调用本机 OCR 命令并直接生成审计计划。

如果已有提交正在进行，扩展会拒绝再次启动提交；等待当前提交结束后再继续。如果浏览器或扩展中断导致状态卡住，6 小时后的旧提交锁会自动过期。

## 安装 uTools 插件

1. 在 uTools 中导入 release 包里的 `cigna-claim-assistant-utools.upx`。
2. 用关键词 `cigna`、`报销`、`理赔` 或 `claim` 打开插件。
3. 选择或拖入一个固定报销文件夹；之后只要把 PDF/图片扔进这个文件夹，再点击扫描即可。也兼容直接拖入一组 PDF/图片文件。填写被保险人、报销理由、治疗国家/地区、理赔类型、就诊类型和付款账户关键词。最早提交服务日期和长期病症最早治疗日期都可留空；拖入文件时只扫描这些指定文件，拖入文件夹时才递归扫描整个目录。
4. 点击 `扫描并导出计划`，插件会调用本地扫描 helper，输出 `outputs/utools-claim-plan.json`。
5. 扫描件没有文本层时，可打开 `扫描时启用本机 OCR`。`OCR 命令` 可留空使用本地扫描 CLI 的 tesseract 兜底，也可填写自己的 OCR wrapper 路径。需要在桌面端先处理超限 PDF 时，可打开 `扫描时压缩超限 PDF`。文件名很乱时，打开 `扫描后自动整理命名副本`，会把识别后的文件复制到 `outputs/organized-claims/服务日期/` 并生成规范文件名；原始文件不会被改名。
6. 可点击 `导出 Chrome 设置`，生成 `outputs/cigna-claim-assistant-chrome-settings-backup.json`；在 Chrome 扩展中点 `导入备份` 选择该文件，可复用 uTools 里的基础设置。导出前页面会显示 Chrome 自动提交设置是否就绪；如果提示 `Chrome 自动提交设置未就绪`，先补齐被保险人、报销理由、治疗国家/地区、理赔类型、就诊类型或付款账户关键词。
7. 点击 `打开 Cigna` 后，在常用 Chrome 中使用已安装的 Chrome 插件完成自动提交。

uTools 插件会保存被保险人、报销理由、治疗国家/地区、理赔类型、就诊类型、付款账户关键词、最早提交服务日期、长期病症最早治疗日期、最近使用的报销目录、OCR 设置和压缩设置，下次打开时自动恢复。

uTools 插件不直接接管 Cigna 登录态，也不替代 Chrome 插件。它的定位是桌面入口、本地扫描和计划导出；真实网页自动提交流程仍在 Chrome 插件里执行。

## 压缩规则

- 单个 PDF 超过 6 MB 会进入压缩计划。
- 同一天附件总量超过 30 MB 会进入压缩计划。
- 自动处理会先压缩，再重新识别，再提交。
- 压缩后仍超目标会停止提交并显示错误；错误会列出最佳压缩后大小和每档 scale/quality 尝试结果，方便改用 SmallPDF 等外部工具压缩后再拖回扩展。
- 提交前还会再次检查单文件 6 MB 和单日报销 30 MB 上限。
- 本地 CLI 压缩和 OCR 可以组合使用：先用 `claims:ocr-sidecars` 或 OCR sidecar 提高日期和类型识别，再让压缩计划处理超限 PDF。

扩展内置压缩会把 PDF 页面重建为 JPEG 页面，是有损压缩，适合扫描件、发票、理赔表。

## 防重防漏规则

- 成功提交过的服务日期会进入本地账本，再次导入同一天会阻塞。
- 同一文件 hash 会跳过。
- `-compressed`、`copy`、原件等逻辑变体会去重，优先保留压缩版；不同日期文件夹里的同名 `scan_1.pdf`、`scan_2.pdf` 会按服务日期/相对文件夹分开，不会互相去重。
- 每次真实提交前都会写入本地批次快照，包含提交指纹、将提交日期、将跳过日期、附件顺序、hash、重复项和是否仍有压缩待处理。失败恢复时先 `导出计划` 或 `导出备份`，用快照核对哪些日期已经发送、哪些日期只是跳过。
- 文件名没有日期但位于日期文件夹中时，例如 `2026-05-14/scan_1.pdf` 和 `2026-05-14/scan_2.pdf`，扩展会按文件夹日期分组，并按 `_1/_2` 推断医疗/理赔表和发票。
- 只有发票/收据会阻塞。
- 只有医疗/理赔表但没有发票/收据会阻塞。
- 只有图片会阻塞。
- 无法识别日期会阻塞。

## 失败恢复

- 某一天提交失败时，后续日期会继续尝试。
- 本地账本只记录真正提交成功的日期。
- 如果浏览器或扩展在提交中断开，先检查 Cigna 页面和本地账本；提交锁 6 小时后会自动过期，之后可重新识别并继续未成功日期。
- 失败后先导出记录或备份，再重新导入文件；已成功日期会被防重规则阻塞，未成功日期仍可继续提交。
- `导出记录` / `导入记录` 只处理防重账本和最近提交状态。
- 重装扩展、换机器或清空 Chrome 数据前，先 `导出备份`；恢复时点 `导入备份` 选择 JSON。备份会恢复基础设置、最近诊断和最近提交快照，并合并账本，不会覆盖当前已有成功提交记录。
- 如果 Cigna 页面结构变化，先点 `检查 Cigna 页面`；结构检查失败时不要继续提交。
- `自动处理并提交` 也会在真正提交前强制做同样的页面结构检查；如果 URL、起始步骤或被保险人卡片不匹配，会停止提交。
- `导出 Cigna 诊断` 可在真实提交前保存页面结构检查 JSON，便于确认扩展看到的是正确 Cigna 标签页和正确被保险人。

## 更新版本

1. 在 `chrome://extensions` 删除旧的 unpacked 扩展，或点击旧扩展的 Reload。
2. 重新 `Load unpacked` 新 release 里的 `extension` 文件夹。
3. 本地账本和基础设置存储在 Chrome 扩展 storage 中；删除扩展可能会清掉这些数据。更新前可先 `导出备份`。
